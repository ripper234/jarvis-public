1. Getting Logs

The first step in any debugging session: get the logs.

Railway Logs (Remote)

If your OpenClaw runs on Railway, pull logs from your local machine:

# Get last 300 lines of Railway logs railway logs --lines 300 > railway-logs.txt # Follow logs in real-time railway logs --follow # Filter for errors only railway logs --lines 500 | grep -i "error\|fail\|crash"
💡 Tip: Save logs to a file first, then search. Real-time log following can miss older context.

On the Container (SSH/Exec)

If you have shell access to the running container:

# OpenClaw status (includes recent errors) openclaw status # Detailed status with all subsystems openclaw status --all # Gateway process log (if redirected) tail -100 /tmp/oc-gateway.log # Check if gateway is running ps aux | grep -i "openclaw\|gateway"

Session Logs

# List recent sessions openclaw sessions list --recent 60 # View a specific session's history openclaw sessions history --key <session-key> # Find session files on disk ls /data/.clawdbot/agents/main/sessions/

2. Status Checks

# Quick health check openclaw status # Full diagnostic openclaw status --all # Check version openclaw --version # Doctor: find config issues openclaw doctor # Doctor with auto-fix openclaw doctor --fix

Key Things to Check

  • Gateway PID: Is the gateway process running?
  • Version match: Does openclaw --version match what you expect?
  • Plugin warnings: Plugin ID mismatches are usually cosmetic in v2026.4.x
  • Disk space: Railway volumes can fill up, especially with session logs

3. Common Issues

Agent Not Responding

  1. Check gateway is running: ps aux | grep gateway
  2. Check Telegram polling: Look for "polling" or "long-poll" in logs
  3. Check for crash loops: railway logs --lines 50
  4. Restart gateway if needed (see Gateway section)

"Plugin ID Mismatch" Warnings

These are cosmetic in v2026.4.x. Safe to ignore unless a plugin actually fails to load.

plugin anthropic: plugin id mismatch (manifest uses "anthropic", entry hints "anthropic-provider") # This is fine. The plugin still works.

Session Store Bloat

# Check session count cat /data/.clawdbot/agents/main/sessions/sessions.json | python3 -c "import json,sys; print(len(json.load(sys.stdin)))" # Check disk usage du -sh /data/.clawdbot/agents/main/sessions/

"openclaw update" Fails

The built-in updater can fail on Railway. Manual update works:

cd /openclaw git fetch --tags git checkout v2026.4.2 # or whatever version CI=true pnpm install --frozen-lockfile # Then restart the gateway
⚠️ Warning: Container packages (pip, apt) are ephemeral on Railway. They survive reboots via the volume but may not survive redeploys.

4. Gateway Troubleshooting

Restart the Gateway

# Option 1: CLI (preferred) openclaw gateway restart # Option 2: Manual (if CLI fails) pkill -f openclaw-gateway nohup node /openclaw/packages/gateway/dist/index.js > /tmp/oc-gateway.log 2>&1 & # Option 3: If running from compiled binary # Check actual process name first: ps aux | grep openclaw
⚠️ Restarting the gateway drops all active sessions. Schedule restarts during quiet hours.

Verify Gateway is Healthy

# RPC probe curl -s http://127.0.0.1:18789/health # Check gateway port ss -tlnp | grep 18789

5. Hooks & Plugins

# List hooks (may hang in some versions) timeout 10 openclaw hooks list # Check hook config directly cat /data/.clawdbot/openclaw.json | python3 -m json.tool | grep -A5 hooks # Enable a hook via config (workaround if CLI hangs) # Edit /data/.clawdbot/openclaw.json, add to hooks.internal.entries # List plugins openclaw plugins list # Inspect a specific plugin openclaw plugins inspect brave

Hook Locations

  • Workspace hooks: <workspace>/hooks/<name>/ (per-agent)
  • Managed hooks: ~/.openclaw/hooks/<name>/ (global)
  • Bundled hooks: Ship with OpenClaw (session-memory, command-logger, etc.)

Available Hook Events

Event When Can Modify?
message:received Inbound message arrives No
message_sending Before outbound delivery Yes (content, cancel)
message:sent After outbound delivery No
command:new User issues /new No

6. Cron Jobs

# List all cron jobs openclaw cron list # Check a specific job openclaw cron info <job-id> # View recent cron runs openclaw cron history --limit 10 # Manually trigger a cron job openclaw cron trigger <job-id>
💡 Lesson from experience: Always set timeoutSeconds and maxConcurrent: 1 on cron jobs. We once had 942 runaway sessions from a cron storm.

7. Telegram Channel

Bot Not Receiving Messages

  1. Check bot token is set: echo $TELEGRAM_BOT_TOKEN | head -c 10
  2. Verify long polling is active in logs
  3. Check allowFrom config isn't blocking your chat ID
  4. Test with BotFather: send /start to your bot

Bot API Limits

  • File download: 20MB max (use Telethon/User API for larger files)
  • Message length: 4096 chars (split or use file attachments)
  • Rate limit: ~30 messages/second to same chat

Finding Your Chat ID

# Check in the config cat /data/.clawdbot/credentials/telegram-allowFrom.json # Or check inbound message metadata in logs railway logs | grep "chat_id\|sender_id"

8. Railway-Specific

Useful Commands

# Get logs railway logs --lines 300 > railway-logs.txt # SSH into the container railway shell # Check environment variables railway variables # Trigger a redeploy railway up

Volume & Disk

# Check disk usage df -h /data # Find large files du -sh /data/* | sort -rh | head -10 # Session logs are often the biggest consumer du -sh /data/.clawdbot/agents/main/sessions/
⚠️ Railway limits: No Docker-in-Docker. Packages installed via apt/pip are ephemeral (survive reboots but not redeploys). Volume data persists.

Manual Update on Railway

# When `openclaw update` fails cd /openclaw git fetch --all --tags git checkout <version-tag> CI=true COREPACK_ENABLE_AUTO_INSTALL=1 pnpm install --frozen-lockfile # Restart gateway after install
