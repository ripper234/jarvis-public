🌪️ The Cron Storm

CRITICAL Apr 1-2, 2026 ✅ RESOLVED

A misconfigured cron job spawned 942 zombie sessions in 15 hours. Jarvis became unresponsive. Messages were lost. The system effectively DDoS'd itself.

5 Whys

1Why was Jarvis unresponsive? 942 active sessions competing for resources.
2Why 942 sessions? Cron jobs spawned new sessions every tick without checking if previous runs completed.
3Why no check? No guardrail: no idempotency, no max-concurrent, no "skip if active" logic.
4Why no guardrail? Cron jobs were added incrementally. Each worked fine in isolation.
5Why no ops policy? We prioritized features over ops hardening. No load testing or failure mode analysis was done.
Root cause: Unbounded concurrency + no lifecycle management for cron-spawned sessions. Classic stateful workload failure. Self-DDoS is the most embarrassing kind of outage.

Fix Applied

  • All 12 cron jobs rebuilt with timeoutSeconds and maxConcurrent: 1
  • Polling interval changed from 60s to 5 min
  • 5-minute health monitoring cron added (sessions, disk, errors, message gaps)
  • 942 zombie sessions cleared

Lessons

  • Batch operations need lifecycle management from day 1
  • "Works in isolation" ≠ "works under load"
  • Every cron job must have explicit timeouts and concurrency limits
  • Build self-monitoring before you need it, not after

📦 Partial Promotion

MEDIUM Apr 2, 2026 ✅ RESOLVED

Staging-to-production promotion missed a file (projects.html). The homepage showed new icons but the projects page showed the old layout.

1Why partial? Manually cherry-picked files instead of full merge.
2Why cherry-pick? git merge failed with directory rename conflicts.
3Why conflicts? Main branch had a staging/ subfolder that confused git.
4Why not diff all files? No promotion checklist. Assumed I knew what changed.
5Why no checklist? Promotion was a new workflow. No automation existed.
Root cause: No automated promotion process. Manual cherry-pick is error-prone when you assume you know the full diff.

Lessons

  • Always diff ALL files between staging and main before promoting
  • Never cherry-pick for promotions -- use full diff
  • Automate repetitive processes before the second time you do them

🔔 Health Alert Spam

HIGH Apr 2, 2026 ✅ RESOLVED

Health check alerts were appended to normal user replies, even after being told to stop. Continued 3+ times after updating the policy file.

1Why did alerts leak into replies? Health info was generated inline during response composition.
2Why inline? No separation between monitoring output and user-facing responses.
3Why did it continue after the policy update? Reading a policy file doesn't guarantee compliance. Behavioral change requires active checking.
4Why no active checking? HOT.md is read but compliance isn't mechanically enforced.
5Why no enforcement? The guardrail system relies on discipline, not code. Discipline failed under session pressure.
Root cause: Policy files are cognitive guardrails, not mechanical ones. Under pressure, reading a rule doesn't mean following it. Trust erosion is the real cost.

Lessons

  • Monitoring output must be architecturally separated from user replies
  • Policy compliance requires active pre-send verification, not just reading
  • Three violations of the same rule is a systemic failure, not a typo
  • Trust is expensive to build and cheap to erode
